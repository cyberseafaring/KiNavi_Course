船联网技术演变

概念边界、发展脉络、AIS 作用、典型形态与趋势

CHAPTER 02

船联网技术演变

船联网把船舶、航道、港口、监管服务和岸端支撑组织成跨域协同系统。理解它的概念边界、演进过程和典型形态，才能读懂后续架构、感知、通信、数据治理与安全机制的真实作用。

边界先明确对象范围，再讨论系统能力
演进从单点数字化走向跨域协同
AIS基础信息层，支撑协同但不覆盖全部能力
趋势绿色航运与智能航海推动能力外延

GO学习目标

本章关注三个核心问题：船联网讨论的对象范围是什么，AIS 等系统为什么能成为海上信息协同的重要基础，以及绿色航运和智能航海提出的业务目标如何转化为系统能力要求。

明确边界

区分船内联网、船船通信、船岸协同和基础设施连接的对象范围。

理解演进

理解定位

把 AIS、e-Navigation、绿色航运、智能航海放回系统语境中理解。

2.1船联网概念边界

概念边界决定后续章节讨论的对象、接口和性能指标。边界不清，系统能力和风险边界就会一起变得模糊。

广义与狭义两种视角

  • 广义视角：把船舶、航道设施、港口系统、监管服务和第三方服务都纳入协同范围
  • 狭义视角：以船端为核心，强调船上数据获取、通信接入、边缘计算和应用闭环

边界取舍带来的差异

边界偏狭义时，更适合讨论采集哪些数据、哪些信息必须可靠、断链时如何运行；边界偏广义时，更适合讨论权限边界、责任边界、共享精度和审计追溯。

对象范围 典型对象 协同信息类型
船端系统 驾驶台、机舱、货舱、能效管理设备 定位、姿态、工况、告警、能耗
航道与海区要素 航标、水文、气象、航道通告 环境状态、风险提示、通航约束
港口与物流系统 靠离泊服务、港口生产、物流协同平台 计划、作业状态、资源调度信息
监管与公共服务 VTS、搜救协调、海事服务 态势监管、预警信息、回溯记录

2.1A协同对象与权限边界图

船端位于中心，周围是航道、港口、监管、第三方服务等协同对象。共享范围越大，权限控制和责任边界越重要。

内圈表示船端闭环关注范围，外圈表示跨域协同带来的权限与责任边界。协同不是无条件的数据互通，而是在规则约束下的信息交换。

2.2发展脉络要点

船联网能力从单点设备数字化、船端集成、岸端服务体系逐步扩展到跨域协同，是一个持续积累的过程。

阶段 1

船端数字化积累

早期重点是信息表达方式改变和计算能力增强，使关键航行信息能够稳定获得、显示并记录。

阶段 2

岸端服务体系发展

航道监测、水文气象、港口生产和监管服务逐步信息化，反过来提升了对船端数据一致性的要求。

阶段 3

协同能力扩展

从单船走向船队、岸端和基础设施协同，信息开始以更结构化方式被交付和复用。

2.2A能力演进阶梯

能力每向外扩展一级，系统价值会增加，但风险和约束也会同步累积。高阶能力依赖低阶能力稳定存在。

每一级能力扩展都带来新的价值，也引入新的风险与约束。高阶能力建立在低阶能力长期稳定运行的基础上。

2.3AIS 基础作用与局限

AIS 之所以重要，是因为它让船舶动态与静态信息能够被交换；AIS 之所以不能等同于船联网，是因为它只覆盖了协同所需信息的一部分，而且存在缺失、延迟和可信度边界。

AIS 的基础作用

  • 提供可交换的船舶动态与静态信息
  • 支撑船船态势共享和岸端监管服务
  • 降低海上信息协同的接口门槛

AIS 的局限

  • 自报属性带来缺失、延迟和被伪造风险
  • 复杂航区可能出现信息拥塞和覆盖受限
  • 单一来源不能被直接等同于真实态势
要点归纳：AIS 更适合作为态势理解的底座信息之一。系统越依赖 AIS，越要显式表达其时效边界和可信度边界。

2.3AAIS 报告间隔说明了什么

AIS 报告间隔会随船舶状态变化而调整。重点在于理解信道资源有限时，系统如何优先传递变化更剧烈的目标信息。

锚泊状态

更新较慢，常见为分钟级，说明状态变化较缓慢。

正常航行

通常为数秒到数十秒，兼顾态势更新与信道占用。

高速转向

更新最频繁，用更高采样密度反映快速变化目标。

2.3Be-Navigation 提供的是协同框架

e-Navigation 的意义不在于某一台设备，而在于把海事信息的收集、融合、交换和呈现放到一致框架下协调，使船端、岸端和监管服务能够在更稳定的接口和语义体系上协同运行。

核心目标

让信息在合适的时间、以合适的形式，到达合适的角色，减少信息过载和接口混乱。

工程含义

协同越多，接口越多，越需要统一语义、稳定流程和清晰权限边界，否则系统将被对接成本和运行风险拖累。

2.4船联网典型形态

在工程实践中，船联网常以若干典型形态出现。不同形态关注的对象、性能约束和失效模式不同，它们通常由内向外逐层叠加。

船内联网

关注设备互联、数据汇聚和本地闭环。

船船交换

关注态势共享、信息延迟风险和互补来源。

船岸协同

关注监管服务、远程运维、缓存补传和审计追溯。

连接基础设施

关注语义一致、服务编排和跨域责任边界。

2.4A典型形态嵌套结构图

四种典型形态可用嵌套结构理解：从内到外逐层扩展，信息共享范围变大，控制与治理要求也随之提高。

船联网形态通常不是互斥选择，而是由内向外逐层叠加的能力体系。共享范围越大，语义一致、权限边界和审计追溯的重要性越高。

2.4B航区差异会改变运行模式

近岸和远洋的通信条件、服务密度和运维窗口不同，因此上报策略、协同方式和风险控制重点也会随之改变。

近岸模式

  • 链路条件相对较好
  • 岸端服务更密集
  • 适合更高频的实时协同

远洋模式

  • 链路更受限，断链更常见
  • 船端闭环能力更重要
  • 更依赖事件化上报和汇总压缩

2.5技术发展趋势

船联网的发展趋势通常沿两条线展开：一条来自绿色航运和智能航海的业务目标，另一条来自海上通信、运维和安全约束的长期存在。

绿色航运

强调能耗、排放和工况数据的可测量、可核查、可追溯，因此更依赖数据治理、对账和权限边界。

智能航海

强调态势理解、风险识别、协同决策和远程支持，因此更依赖多源信息协同、时间一致性和可降级运行。

2.5A业务目标到基础能力的映射

把业务目标落到系统设计时，应先明确需要什么数据特征和什么基础能力，再讨论具体技术实现方式。

业务目标 需要的数据特征 对应基础能力
能效分析与核查 长周期完整、来源可追溯、口径一致 数据治理、对账、审计
远程运维 状态连续、过程留痕、版本可回滚 缓存补传、配置管理、审计追溯
协同决策支持 时序一致、共享边界清晰、过程可复盘 接口一致、服务编排、日志与回放

KEY术语速览

AIS / VTS

分别对应自动识别系统和船舶交通服务，是海上协同的基础信息系统。

e-Navigation

海事信息服务的组织框架，强调一致语义、一致接口和可持续协同。

绿色航运 / 智能航海

两个重要业务牵引方向，分别推动数据治理和协同决策能力增强。

思考题

  1. 为什么说船联网讨论的关键是协同能力，而不只是通信接入能力？
  2. 若仅依赖 AIS 数据构建航行态势，会在哪些条件下出现误判风险？
  3. 为什么绿色航运相关应用会更强调可追溯、口径一致和对账能力？
  4. 为什么智能航海相关应用必须强调可降级运行，而不能只强调能力增强？
  5. 近岸和远洋两种运行模式下，系统在数据上报和风险控制上的重点有何差异？
